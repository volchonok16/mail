# Свой SMTP‑сервер: настройка и DNS

В проекте реализован **собственный SMTP** без зависимости от внешних сервисов (SendGrid и т.п.) для приёма и отправки почты. В `docker-compose.yml` уже включён режим «свой сервер»: `SMTP_RELAY_ENABLED=false`, отправка идёт напрямую по MX.

---

## Записи DNS для домена alexol.io

Замените **`ВАШ_IP`** на реальный IP вашего сервера (VPS), где крутится Docker с backend.

### 1. A-запись (хост для почты)

| Тип | Имя (Host) | Значение (Points to) | TTL |
|-----|-------------|----------------------|-----|
| A   | `mail`      | `ВАШ_IP`             | 300 |

Итог: `mail.alexol.io` → ваш сервер. Если почта будет с того же IP, что и сайт alexol.io, можно использовать уже существующую A-запись для `@` (корень домена).

### 2. MX (куда принимать письма для @alexol.io)

| Тип | Имя (Host) | Значение                    | Приоритет | TTL |
|-----|------------|-----------------------------|-----------|-----|
| MX  | `@`        | `mail.alexol.io`            | 10        | 300 |

(В панели DNS «Имя» может быть пустым или `@` — это корень домена alexol.io.)

### 3. SPF (разрешённые серверы для отправки)

| Тип | Имя (Host) | Значение | TTL |
|-----|------------|----------|-----|
| TXT | `@`        | `v=spf1 ip4:ВАШ_IP -all` | 300 |

Либо, если у вас уже есть A-запись для `mail.alexol.io`:

| Тип | Имя | Значение |
|-----|-----|----------|
| TXT | `@` | `v=spf1 a:mail.alexol.io -all` |

Одна запись SPF на домен — достаточно (либо `ip4:ВАШ_IP`, либо `a:mail.alexol.io`).

### 4. DMARC (рекомендация по проверке писем, по желанию)

| Тип | Имя (Host) | Значение | TTL |
|-----|-------------|----------|-----|
| TXT | `_dmarc`    | `v=DMARC1; p=none; rua=mailto:admin@alexol.io` | 300 |

После сохранения записей подождите 5–15 минут (иногда до часа), затем проверьте:

- MX: `nslookup -type=mx alexol.io`
- SPF: `nslookup -type=txt alexol.io`
- A: `nslookup mail.alexol.io`

## Что есть сейчас

- **Порт 25** — приём почты (без аутентификации). Другие сервера и локальная доставка шлют сюда.
- **Порт 587** — Submission: отправка из почтовых клиентов с **логином/паролем** (учётная запись из БД) и **STARTTLS**.
- **Отправка на внешние адреса** — либо через SMTP Relay (SendGrid и др.), либо **напрямую по MX** (без relay).

Relay в `docker-compose.yml` уже отключён (`SMTP_RELAY_ENABLED=false`), поэтому отправка на любые адреса идёт **напрямую** по MX. Чтобы письма с @alexol.io не попадали в спам и не отвергались, добавьте DNS-записи выше.

## DNS для своего домена

### 1. A/AAAA записи

Укажите ваш сервер как хост для домена почты, например:

- `mail.ваш-домен.ru` → IP вашего VPS/сервера  
- Или тот же IP для корня домена, если почта идёт с `@ваш-домен.ru`.

### 2. MX запись

Указывает, какой хост принимает почту для домена:

- Для `ваш-домен.ru`: MX на `mail.ваш-домен.ru` (или на хост с вашим приложением), приоритет например `10`.

Пример (в панели DNS хостинга):

| Тип | Имя  | Значение           | Приоритет |
|-----|------|--------------------|-----------|
| MX  | @    | mail.ваш-домен.ru  | 10        |

### 3. SPF (Sender Policy Framework)

Говорит другим серверам: «письма от этого домена могут отправляться только с моих серверов».

Добавьте TXT-запись:

- Имя: `@` (или корень домена)
- Значение:  
  `v=spf1 ip4:ВАШ_IP_СЕРВЕРА -all`  
  или, если отправляете только с одного хоста:  
  `v=spf1 a:mail.ваш-домен.ru -all`

Замените `ВАШ_IP_СЕРВЕРА` на реальный IP или используйте `a:mail.ваш-домен.ru`, если A-запись уже указывает на этот сервер.

### 4. DKIM (подпись писем)

DKIM подписывает исходящие письма ключом домена. Сейчас в коде **нет подписи DKIM** — её нужно добавить на этапе отправки (в backend при формировании письма). После добавления:

- Сгенерируйте ключ (приватный — на сервере, публичный — в DNS).
- Добавьте TXT-запись с публичным ключом, имя обычно вида `selector._domainkey.ваш-домен.ru`.

Подробности зависят от того, как вы реализуете подпись (библиотека, формат заголовка).

### 5. DMARC (опционально)

Рекомендация получателям, что делать с письмами, не прошедшими SPF/DKIM:

- Имя: `_dmarc`
- Значение:  
  `v=DMARC1; p=none; rua=mailto:admin@ваш-домен.ru`  
  (для начала `p=none` — только отчёты, без отбрасывания).

## TLS для порта 587

Для Submission (587) желательно использовать TLS (STARTTLS):

- **Сертификаты заданы** — в конфиге укажите:
  ```env
  SMTP_TLS_CERT_FILE=/path/to/cert.pem
  SMTP_TLS_KEY_FILE=/path/to/key.pem
  ```
  В Docker можно смонтировать том с сертификатами и указать пути внутри контейнера.

- **Сертификаты не заданы** — при старте backend автоматически создаётся **временный self-signed** сертификат (если в системе есть `openssl`). Для разработки этого достаточно; клиенты могут предупреждать о ненадёжном сертификате. Для продакшена лучше использовать Let's Encrypt (например, certbot) и указать выданные им `fullchain.pem` и `privkey.pem`.

## Проверка

- **Приём (25):** с другого сервера или с хоста:
  ```bash
  swaks -to user@ваш-домен.ru -from test@example.com -server localhost:25
  ```
- **Submission (587):** в почтовом клиенте (Outlook, Thunderbird и т.д.):
  - Сервер исходящей почты: ваш хост, порт **587**
  - Шифрование: **STARTTLS**
  - Логин/пароль: email и пароль пользователя из вашего приложения (БД).

После настройки DNS (MX, SPF, при необходимости DKIM/DMARC) отправка «напрямую» по MX будет работать без SendGrid и других relay-сервисов.
